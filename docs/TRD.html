1) Executive Summary

ExampleHR’s time-off service is a NestJS microservice that manages leave requests while treating HCM as the single source of truth for balances. The local SQLite database is intentionally a cache + operational ledger, enabling strict workflow control (state transitions, idempotency, retries, auditability) without inventing authoritative balance values.

The design prioritizes correctness under failure and concurrency: all writes are serialized by SQLite BEGIN IMMEDIATE, HCM writes are mediated by an outbox worker, drift is corrected through real-time fetch + batch sync + reconciliation, and all critical transitions are append-only logged. This architecture deliberately trades horizontal scalability for deterministic behavior and testable resilience.

3) System Invariants / Rules (R1-R15)

  1. R1: HCM is sole balance authority.
  2. R2: No local approval without HCM-confirmed deduction.
  3. R3: Idempotency required on every mutating path.
  4. R4: SQLite serializes all writes globally; handle SQLITE_BUSY as 503.
  5. R5: Stale (>5m) balance cannot drive approval; re-fetch required.
  6. R6: Batch updates apply only if incoming HCM timestamp is strictly newer.
  7. R7: Strict state machine; invalid transitions throw exceptions.
  8. R8: HCM calls use 8s timeout with explicit 409/4xx/5xx handling.
  9. R9: All HCM writes must flow through outbox.
  10. R10: Audit logs are immutable append-only.
  11. R11: No silent failures; all workers must log start/success/failure structurally.
  12. R12: Request dimensions are immutable after creation.
  13. R13: Reconciliation compares total_days and adjusted used_days; auto-correct gated.
  14. R14: Coverage gates on service layer, state machine, outbox processor, and API endpoint tests.
  15. R15: Canonical retry/backoff: max 3 attempts, delays 2s/4s/8s.

5) API Contracts

GET /balances/:employeeId/:locationId/:leaveType

Returns computed availableDays = totalDays - usedDays - pendingDays. TTL refresh after 5 minutes.

POST /time-off/requests

Creates request (202) with idempotency and async outbox deduction flow.

GET /time-off/requests/:requestId / GET /time-off/requests

Single request read and paginated list by employee.

POST /time-off/requests/:requestId/cancel

SUBMITTED → CANCELLED directly, APPROVED → CANCELLING with outbox reverse.

POST /sync/batch/balances and GET /sync/reconciliation/status

Batch apply endpoint and latest reconciliation summary endpoint.

GET /health

Returns status, HCM reachability, outbox pending count, last batch and reconciliation timestamps.

6) Challenges & Decisions

C1-C3: Consistency, independent HCM updates, and write serialization
Cache-as-ledger design, push+pull+TTL sync, and SQLite global writer lock model were chosen for determinism.
C4-C6: Idempotency, ambiguous HCM outcomes, and stale batch protection
Claim-before-execute idempotency, 409-as-success semantics, and per-record R6 timestamp checks prevent duplicate side effects.
C7-C9: HCM downtime, state mismatch, and defensive validation
Fresh cache can proceed, stale cache blocks writes, outbox crash recovery resets stuck processing, local validation remains mandatory.
C10-C13: fractional leaves, backdated requests, cancellation reversal, first-use balance creation
0.5-day granularity, controlled retroactive behavior, explicit CANCELLING state, and safe first-row creation under lock.

8) Concurrency & Idempotency

  • SQLite WAL + BEGIN IMMEDIATE serialize all writers, including different employees.
  • withBalanceLock is the only legal balance mutation entry-point.
  • Idempotency: interceptor + DB uniqueness + HCM externalRef uniqueness.
  • Outbox claim/reset protects against duplicate and stuck processing records.
availableDays = totalDays - usedDays - pendingDays

11) Mock HCM Design

Mock HCM is a full NestJS service, not a static stub, with independent SQLite state and control-plane APIs.

Key behaviors

  • Deduct/reverse rules enforce externalRef idempotency and transactional effects.
  • Chaos controls: timeout, 500, 409, slow, silent_success, invalid_validation.
  • Batch endpoint supports snapshot + cursor pagination with expiring snapshots.
  • Control APIs enable deterministic tests: set balances, inject drift, advance HCM clock, inspect call logs, reset state.

Failure scenarios covered

FS-1..FS-10 including silent success, network cuts, stale batch timestamps, reversal failures, and validation inconsistency modes.
