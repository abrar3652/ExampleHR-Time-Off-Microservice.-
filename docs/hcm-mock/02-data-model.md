# HCM Mock — 02: Data Model

> Mock HCM uses its own SQLite database (`hcm-mock.db`), entirely separate from ReadyOn's database.
> This ensures there is NO shared state between the two systems.

---

## Table: `hcm_balance`

The authoritative balance as HCM knows it.

```sql
CREATE TABLE hcm_balance (
  id                  TEXT PRIMARY KEY,
  employee_id         TEXT NOT NULL,
  location_id         TEXT NOT NULL,
  leave_type          TEXT NOT NULL,
  total_days          REAL NOT NULL,
  used_days           REAL NOT NULL DEFAULT 0,
  last_updated_at     TEXT NOT NULL,   -- HCM's internal clock (can be advanced via control API)
  created_at          TEXT NOT NULL,

  UNIQUE(employee_id, location_id, leave_type)
);
```

---

## Table: `hcm_transaction`

Every deduction or reversal applied by HCM.

```sql
CREATE TABLE hcm_transaction (
  id                  TEXT PRIMARY KEY,           -- hcm-txn-uuid (generated by HCM)
  external_ref        TEXT NOT NULL UNIQUE,       -- externalRef from ReadyOn (idempotency key)
  employee_id         TEXT NOT NULL,
  location_id         TEXT NOT NULL,
  leave_type          TEXT NOT NULL,
  transaction_type    TEXT NOT NULL,              -- DEDUCT | REVERSE
  days                REAL NOT NULL,
  start_date          TEXT,
  end_date            TEXT,
  status              TEXT NOT NULL,              -- APPLIED | REVERSED | SILENT_FAILED
  reversed_by         TEXT,                       -- transaction id of reversal
  created_at          TEXT NOT NULL
);

CREATE INDEX idx_hcm_txn_ext_ref ON hcm_transaction(external_ref);
CREATE INDEX idx_hcm_txn_employee ON hcm_transaction(employee_id);
```

**`SILENT_FAILED`** — a transaction recorded by HCM but NOT applied to `hcm_balance`. Used to simulate the "returns 200 but doesn't actually apply" failure mode. This allows ReadyOn's defensive reconciliation to be tested.

## Table: `hcm_chaos_config`

Current chaos behavior configuration (set via `/__control/behavior`).

```sql
CREATE TABLE hcm_chaos_config (
  id                  TEXT PRIMARY KEY DEFAULT 'singleton',
  config              TEXT NOT NULL DEFAULT '{}'  -- JSON: { endpoint -> ChaosRule }
);
```

**ChaosRule JSON structure:**
```json
{
  "balance_get": {
    "behavior": "timeout",
    "remaining_count": 2,
    "delay_ms": 0
  },
  "deduct": {
    "behavior": "silent_success",
    "remaining_count": -1,
    "delay_ms": 2000
  }
}
```

## Table: `hcm_call_log`

Append-only log of all incoming calls (for test assertions).

```sql
CREATE TABLE hcm_call_log (
  id                  TEXT PRIMARY KEY,
  endpoint            TEXT NOT NULL,
  method              TEXT NOT NULL,
  request_body        TEXT,
  response_status     INTEGER NOT NULL,
  response_body       TEXT,
  chaos_applied       TEXT,             -- null or the chaos behavior applied
  duration_ms         INTEGER NOT NULL,
  called_at           TEXT NOT NULL
);
```

---

## Table: `hcm_internal_clock`

Tracks HCM's simulated clock offset for controlling `last_updated_at` timestamps.

```sql
CREATE TABLE hcm_internal_clock (
  id                  TEXT PRIMARY KEY DEFAULT 'singleton',
  offset_ms           INTEGER NOT NULL DEFAULT 0  -- added to now() when generating timestamps
);
```

**Usage:**
```typescript
function hcmNow(): string {
  const offset = clockRepo.getOffset();
  return new Date(Date.now() + offset).toISOString();
}
```

This allows tests to produce records with future or past `last_updated_at` values to test R6 (batch ordering) without waiting for real time to pass.
