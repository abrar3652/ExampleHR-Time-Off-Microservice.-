# 02 — Data Model

> All tables use SQLite via TypeORM (`better-sqlite3` driver). All timestamps are UTC ISO-8601 strings.
> Primary keys are UUIDs (v4) generated by the application layer, not DB auto-increment.
> `available_days` is computed in the **service layer only** — it is NOT a database column.

---

## Table: `balance`

Stores the local cached view of HCM balances, scoped per employee + location + leave_type.

```sql
CREATE TABLE balance (
  id                  TEXT PRIMARY KEY,          -- UUID v4
  employee_id         TEXT NOT NULL,
  location_id         TEXT NOT NULL,
  leave_type          TEXT NOT NULL,             -- ENUM: ANNUAL | SICK | UNPAID | MATERNITY | PATERNITY
  total_days          REAL NOT NULL,             -- supports 0.5 increments
  used_days           REAL NOT NULL DEFAULT 0,
  pending_days        REAL NOT NULL DEFAULT 0,   -- sum of days for SUBMITTED/PENDING_HCM requests
  hcm_last_updated_at TEXT NOT NULL,             -- timestamp from HCM payload (used in R6)
  synced_at           TEXT NOT NULL,             -- when WE last fetched/received this from HCM
  created_at          TEXT NOT NULL,
  updated_at          TEXT NOT NULL,

  UNIQUE(employee_id, location_id, leave_type)
);
```

**`available_days` is computed in TypeScript, never stored:**
```typescript
// In BalanceService — the only place this calculation exists
function computeAvailableDays(balance: Balance): number {
  return balance.totalDays - balance.usedDays - balance.pendingDays;
}
```

**Invariants:**
- `computeAvailableDays(balance)` MUST be checked before every deduction. If result < 0 after a successful HCM approval, log `CRITICAL_BALANCE_ANOMALY` and trigger `HCM_REVERSE`.
- `pending_days` is maintained by the service layer (incremented on SUBMITTED, decremented on APPROVED/REJECTED/CANCELLED/FAILED).
- `used_days` is set to `hcmResponse.newUsedDays` (authoritative HCM value) when HCM confirms a deduction — never computed as a local delta.
- `hcm_last_updated_at` is set from the HCM API response body — NEVER from local clock.
- When HCM confirms approval, `hcm_last_updated_at` and `synced_at` are updated in the same transaction.

## Table: `time_off_request`

Lifecycle record for a single time-off request.

```sql
CREATE TABLE time_off_request (
  id                    TEXT PRIMARY KEY,           -- UUID v4
  idempotency_key       TEXT NOT NULL UNIQUE,       -- client-supplied; enforces R3
  employee_id           TEXT NOT NULL,
  location_id           TEXT NOT NULL,
  leave_type            TEXT NOT NULL,
  start_date            TEXT NOT NULL,              -- YYYY-MM-DD
  end_date              TEXT NOT NULL,              -- YYYY-MM-DD
  days_requested        REAL NOT NULL,              -- must be > 0, supports 0.5
  state                 TEXT NOT NULL,              -- ENUM: see R7
  last_outbox_event     TEXT,                       -- ENUM: HCM_DEDUCT | HCM_REVERSE — tracks which operation is in flight
  hcm_external_ref      TEXT,                       -- stable ref sent to HCM for deductions (= request.id)
  hcm_transaction_id    TEXT,                       -- HCM's own transaction ID, returned on successful deduction; required for reversal
  hcm_response_code     INTEGER,                    -- last HCM HTTP status
  hcm_response_body     TEXT,                       -- raw HCM response (JSON string)
  rejection_reason      TEXT,
  failure_reason        TEXT,                       -- ENUM: DEDUCTION_FAILED | REVERSAL_FAILED — disambiguates FAILED state
  retry_count           INTEGER NOT NULL DEFAULT 0,
  created_by            TEXT NOT NULL,              -- employee_id of requester
  approved_by           TEXT,                       -- manager employee_id
  created_at            TEXT NOT NULL,
  updated_at            TEXT NOT NULL
);

CREATE INDEX idx_tor_employee   ON time_off_request(employee_id);
CREATE INDEX idx_tor_state      ON time_off_request(state);
CREATE INDEX idx_tor_idempotency ON time_off_request(idempotency_key);
```

**Invariants:**
- `idempotency_key` is UNIQUE — duplicate inserts fail at DB level.
- `days_requested` = business days between `start_date` and `end_date` (computed by service layer, validated before insert).
- `hcm_external_ref` = `request.id` — set at creation, never changed.
- `hcm_transaction_id` — populated by OutboxProcessor when HCM returns 200 on a DEDUCT call. Required in the outbox payload for any subsequent HCM_REVERSE operation.
- `last_outbox_event` — set whenever an outbox record is created for this request. Tells any reader what operation is currently in-flight. Cleared (set NULL) when outbox reaches DONE or FAILED.
- `failure_reason` — set on every `→ FAILED` transition: `DEDUCTION_FAILED` when the deduct outbox exhausted retries; `REVERSAL_FAILED` when the reverse outbox exhausted retries.
- `state` transitions enforced by `RequestStateMachine` (R7).

## Table: `outbox`

Transactional outbox for async HCM writes.

```sql
CREATE TABLE outbox (
  id                  TEXT PRIMARY KEY,           -- UUID v4
  event_type          TEXT NOT NULL,              -- ENUM: HCM_DEDUCT | HCM_REVERSE
  payload             TEXT NOT NULL,              -- JSON string
  request_id          TEXT NOT NULL REFERENCES time_off_request(id),
  status              TEXT NOT NULL DEFAULT 'PENDING',  -- ENUM: PENDING | PROCESSING | DONE | FAILED
  attempts            INTEGER NOT NULL DEFAULT 0,  -- starts at 0; incremented before each execution attempt
  last_attempted_at   TEXT,
  last_error          TEXT,
  created_at          TEXT NOT NULL,
  process_after       TEXT NOT NULL               -- UTC datetime; supports delayed retry
);

CREATE INDEX idx_outbox_pending ON outbox(status, process_after)
  WHERE status = 'PENDING';

-- Enforces at most one active (non-DONE) outbox record per request.
-- Prevents orphaned records on FAILED→SUBMITTED retry.
CREATE UNIQUE INDEX idx_outbox_active ON outbox(request_id)
  WHERE status != 'DONE';
```

**Invariants:**
- Exactly one active (non-DONE) `outbox` record per request at any time, enforced by `idx_outbox_active`.
- `outbox` record and the triggering `time_off_request` state change are written in the SAME `BEGIN IMMEDIATE` transaction.
- `PROCESSING` records older than 30 seconds are reset to `PENDING` by the worker watchdog on every poll tick.
- `attempts` is incremented **before** the HCM call executes. After increment, if `attempts > MAX_ATTEMPTS (3)`, mark FAILED immediately without calling HCM. This ensures exactly 3 HCM call attempts (attempts=1, 2, 3).
- On `FAILED → SUBMITTED` retry: reset this record to `status=PENDING, attempts=0, last_error=NULL, process_after=NOW` rather than inserting a new record. The unique index enforces this is safe.

## Table: `request_audit_log`

Immutable append-only log of every state transition.

```sql
CREATE TABLE request_audit_log (

CREATE INDEX idx_ral_request ON request_audit_log(request_id);
```

## Table: `balance_change_log`

Immutable log of every balance field mutation. One row per field changed per operation.

```sql
CREATE TABLE balance_change_log (
  id                  TEXT PRIMARY KEY,
  balance_id          TEXT NOT NULL REFERENCES balance(id),
  employee_id         TEXT NOT NULL,
  location_id         TEXT NOT NULL,
  leave_type          TEXT NOT NULL,
  field_changed       TEXT NOT NULL,              -- ENUM: used_days | pending_days | total_days
  old_value           REAL NOT NULL,
  new_value           REAL NOT NULL,
  delta               REAL NOT NULL,              -- new_value - old_value
  source              TEXT NOT NULL,              -- ENUM: REQUEST | BATCH_SYNC | REAL_TIME_SYNC | AUTO_RECONCILE | MANUAL
  source_ref          TEXT,                       -- request_id or batch_id or reconciliation_id
  hcm_timestamp       TEXT,                       -- hcm_last_updated_at at time of change
  created_at          TEXT NOT NULL
);

CREATE INDEX idx_bcl_balance  ON balance_change_log(balance_id);
CREATE INDEX idx_bcl_employee ON balance_change_log(employee_id);
```

**Rule:** If an operation changes both `total_days` and `used_days`, two rows are written — one per field. The `source_ref` is the same on both, enabling reconstruction of the full operation.

## Table: `reconciliation_log`

```sql
CREATE TABLE reconciliation_log (
  id                  TEXT PRIMARY KEY,
  run_id              TEXT NOT NULL,              -- groups all entries from one reconciliation run
  employee_id         TEXT NOT NULL,
  location_id         TEXT NOT NULL,
  leave_type          TEXT NOT NULL,
  drift_field         TEXT NOT NULL,              -- ENUM: total_days | used_days — which field drifted
  local_value         REAL NOT NULL,              -- raw local value (not adjusted for pending)
  hcm_value           REAL NOT NULL,
  adjusted_local      REAL NOT NULL,              -- local_value + pending_days (for used_days comparison only)
  drift               REAL NOT NULL,              -- hcm_value - adjusted_local
  resolved            INTEGER NOT NULL DEFAULT 0, -- 0 | 1
  resolution          TEXT,                       -- AUTO_CORRECTED | MANUAL_REVIEW | IGNORED
  resolved_at         TEXT,                       -- populated when resolved = 1
  created_at          TEXT NOT NULL
);
```

## Table: `sync_checkpoint`

Tracks last successful batch sync position.

```sql
CREATE TABLE sync_checkpoint (
  id                  TEXT PRIMARY KEY DEFAULT 'singleton',  -- always one row
  last_batch_id       TEXT,
  last_batch_at       TEXT,                       -- HCM's generatedAt, NOT local now()
  last_record_count   INTEGER,
  updated_at          TEXT NOT NULL
);
```

**Rule:** `last_batch_at` stores the HCM batch's `generatedAt` timestamp (HCM clock), not the local server time. This ensures the `since` parameter on next pull uses the same clock as HCM's `last_updated_at` values. Updated only after all pages of a batch complete successfully.

## Table: `idempotency_record`

Stores in-progress and completed idempotent operations.

```sql
CREATE TABLE idempotency_record (
  idempotency_key     TEXT PRIMARY KEY,
  status              TEXT NOT NULL DEFAULT 'IN_PROGRESS',  -- ENUM: IN_PROGRESS | COMPLETE
  response_status     INTEGER,                    -- NULL while IN_PROGRESS
  response_body       TEXT,                       -- NULL while IN_PROGRESS; JSON string when COMPLETE
  expires_at          TEXT NOT NULL,              -- 24h from creation
  created_at          TEXT NOT NULL
);

CREATE INDEX idx_ir_expires ON idempotency_record(expires_at);
```

**Invariants:**
- Row is inserted as `IN_PROGRESS` **before** the business operation runs, inside the same `BEGIN IMMEDIATE` transaction as the business operation itself. This is the claim step.
- If the business operation fails (exception thrown), the `IN_PROGRESS` row is deleted in the rollback.
- After business operation succeeds, the row is updated to `COMPLETE` with response data.
- Concurrent requests with the same key: the second `INSERT` fails with `UNIQUE` constraint violation → return 409 `IDEMPOTENCY_IN_PROGRESS` (client should retry with GET).
- Expired records (`expires_at < now()`) are deleted by a nightly `@Cron` job.
